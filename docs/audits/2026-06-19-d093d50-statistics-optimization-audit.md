# d093d50 故障统计优化提交审计

审计对象：`d093d503bc80a4e3cab042163041bd14757069ba`

提交时间：2026-06-17 10:20:59 +08:00

提交说明：`故障统计优化；割接保存bug修改`

## 审计结论

该提交不是单纯的性能优化。它同时修改了统计接口契约、业务卡片默认范围、明细加载方式、筛选执行位置、分页和重复故障展示逻辑，并删除或改写了原有行为保护测试。

应保留的优化包括重复故障检测集中化、历史汇总缓存思路、汇总与明细接口拆分、数据库索引方向，以及割接序列化器修正。但统计优化必须以原有统计口径和展示行为为基线重新实现，不能继续以删除原有测试或改变默认结果集来换取响应速度。

## 高严重度问题

### 1. 明细接口拆分时没有完整迁移原有筛选语义

提交前，物理故障明细在浏览器完整数据集上执行筛选；提交后，前端把筛选参数传给新的 `FaultStatisticsDetailsAPI`。但是新接口只实现了：

- `category`
- `resource_type`
- `reason`
- `is_long`
- `duration_bucket`
- `scope`
- `province`
- `ordering`

原有界面仍会发出但接口未实现的条件包括：

- `source_group`
- `is_repeat`
- `occurrence_period`
- `cause_group`
- `duration_min`
- `duration_max`
- `duration_histogram_bucket`
- 图例排除参数 `exclude_resource_type`、`exclude_province`、`exclude_reason`

此外，界面传入的 `category` 值是中文显示值，例如 `光缆中断`，接口却直接用于数据库枚举字段过滤，无法匹配内部枚举值。

这不是性能差异，而是下钻统计逻辑失效。当前代码仍有部分残留：前端继续传递这些条件，而后端仍未完整处理。

### 2. 分页改变了重复故障排序的数据边界

提交将完整明细改为每页 25 条，并仅在当前页数据上检测和分组重复故障。由此产生：

- 重复组被分页切断；
- 当前页之外的同组故障无法参与排序；
- 前序周期关联故障不再进入完整排序数据集；
- “按重复排序”退化为当前页局部排序。

该问题已由 `b4d3306` 取消明细分页并恢复完整结果集排序。

### 3. 业务类型参数使用显示值过滤数据库枚举

提交后的前端向业务明细接口发送 `裸纤业务` / `电路业务`，后端直接执行 `service_type=<中文显示值>`，导致裸纤和电路明细始终为空。

该问题已由 `0f5551f` 修复为发送 `bare_fiber` / `circuit`，后端同时兼容中文别名。

### 4. “全部裸纤业务”默认数据被性能优化裁掉

提交前会初始化全部裸纤业务；提交后只构造当前周期受影响的业务。前端“全部业务”切换仍只操作已经返回的列表，因此无法显示无故障业务。

该问题已由 `071d3d1` 改为仅在用户切换“全部业务”时显式请求全量裸纤业务。

## 中严重度问题

### 5. 历史汇总缓存键遗漏 `calendar_year` 和 `calendar_month`

`FaultStatisticsDataAPI` 的响应包含由 `calendar_year`、`calendar_month` 决定的子公司月历数据，但缓存键只包含统计周期和省份：

```text
fault-summary:{filter_type}:{start_date}_{end_date}:{provinces_hash}
```

相同统计周期下切换日历年份或月份时，会命中前一次请求的缓存并返回错误的月历内容。该问题当前仍存在。

### 6. 索引只写入模型，没有生成迁移

提交在 `OtnFault` 和 `OtnFaultImpact` 的 `Meta.indexes` 中增加了三个索引，但提交中没有迁移文件，当前迁移目录也没有这些索引名。

因此：

- 已部署数据库不会创建这些索引；
- 预期的查询性能收益不存在；
- 模型状态与迁移状态不一致，后续 `makemigrations` 会产生补偿迁移。

### 7. 原有行为测试被删除或改写，掩盖回归

提交删除了子公司明细缓存、按时间/重复排序、后端子公司范围、前序重复记录和本地筛选等断言，并把“全部业务”和业务明细测试改成新的分页接口结构。

这些测试没有被等价的接口行为测试替代，导致接口参数缺失、中文枚举不匹配、排序边界变化可以在测试通过的情况下进入主分支。

### 8. 子公司“有效历时”条件未迁移

新接口最初没有处理 `is_valid_duration=true`，导致标签切换只改变提示，不改变结果。

该问题已由 `4390e1a` 修复。

## 当前性能结构风险

取消分页后，当前页面加载汇总数据时会立即调用物理明细和子公司明细接口，两者都返回完整结果；业务统计加载后也会立即请求裸纤和电路完整明细。

这意味着主接口虽然瘦身，但总请求量可能变为：

```text
汇总接口 + 物理全量明细 + 子公司全量明细
业务汇总接口 + 裸纤全量明细 + 电路全量明细
```

正确方向应是保留独立明细接口，但只在对应 Tab 可见或用户首次下钻时加载，并按统计条件缓存完整明细。这样既能保留完整排序和原有展示逻辑，也不会在每次周期切换时加载所有明细。

## 缓存失效风险

统计缓存版本目前监听：

- `OtnFault`
- `OtnFaultImpact`
- `BareFiberService`
- `CircuitService`
- 故障 Z 端 M2M
- 影响业务 Z 端 M2M

但统计结果还依赖站点名称、省份、处理单位等关联对象。直接修改这些关联对象不会递增统计缓存版本，已结束周期可能在 12 小时内继续展示旧值。

## 割接保存修改

割接部分的修改是独立且合理的：

- `CutoverTaskSerializer` 补充模型中实际存在的 `cutover_type`；
- 删除模型中不存在的 `related_customers`。

这部分属于割接 API 保存修复，不应与故障统计性能重构混在同一个提交中。

## 建议处理顺序

1. 补齐并测试所有原有下钻筛选参数，统一显示值与数据库枚举值映射。
2. 修复历史缓存键，加入 `calendar_year` 和 `calendar_month`。
3. 为三个统计索引生成正式迁移并执行迁移验证。
4. 改为按 Tab/首次下钻懒加载完整明细，避免每次周期切换加载四张全量表。
5. 补充缓存失效覆盖或缩短关联数据变化场景的缓存周期。
6. 恢复“原逻辑基线测试”，性能重构测试必须同时证明统计结果和展示行为不变。
