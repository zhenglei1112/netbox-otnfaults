# 故障统计明细取消分页与重复排序恢复设计

## 背景

提交 `d093d50` 将故障统计页下方明细拆为分页接口。分页只向前端提供当前页数据，导致原有“按重复排序”无法在完整数据集上识别、分组和展示 60 天内关联故障。同时，物理故障、子公司、裸纤业务和电路业务四张明细表增加的分页控件不符合当前使用需求。

## 目标

- 取消物理故障、子公司统计、裸纤业务和电路业务四张明细表的分页。
- 每次筛选请求一次返回全部匹配明细。
- 物理故障和子公司明细恢复原有“按时间 / 按重复”前端排序。
- “按时间”仅展示当前统计周期内的数据。
- “按重复”展示完整重复组，包含与当前周期故障关联的前序周期故障。
- 裸纤业务和电路业务继续支持点击业务卡片过滤明细。
- 保留现有图表下钻、有效历时、长时故障、省份和分类等服务端筛选。

## 后端接口

### 故障明细

`FaultStatisticsDetailsAPI` 不再读取 `page` 和 `per_page`，直接返回：

```json
{
  "results": []
}
```

查询流程：

1. 根据统计周期和下钻参数筛选当前周期故障。
2. 在“按重复”需要的数据语义下，为当前筛选结果查询前 60 天内可能关联的光缆故障。
3. 使用现有重复故障检测逻辑标记当前周期和关联历史故障。
4. 当前周期记录返回 `in_period=true`，关联历史记录返回 `in_period=false`。
5. 子公司 `scope=branch_company` 对当前记录和关联历史记录应用相同的子公司归属及维护单位排除规则。

服务端不负责最终重复组排序，只提供完整且带标记的数据集。

### 业务明细

`ServiceStatisticsDetailsAPI` 同样移除分页切片和 `pagination` 响应，返回全部符合统计周期、业务类型和业务卡片过滤条件的影响记录。

## 前端数据流

移除四张明细表的：

- 页码导航
- “显示 X-Y 共 Z”
- 每页数量下拉框
- 页码、每页数量和总数状态变量
- 通用分页渲染调用

物理故障请求成功后将完整结果保存到 `currentAllDetails`；子公司请求成功后保存到 `currentBranchCompanyDetails`。`renderDetailsTable()` 和 `renderBranchCompanyDetailsTable()` 恢复为对本地完整结果执行：

1. 当前筛选结果展示。
2. `assignRepeatGroupColors()`。
3. 根据选中的排序模式调用 `sortDetailRows()`。
4. 重新计算过滤结果的局部统计。
5. 渲染完整表格。

业务明细直接渲染完整结果；点击业务卡片时重新请求对应业务的完整明细，清除过滤时重新请求该业务类型的完整明细。

## 重复排序语义

- 判断维度：A 端站点相同，Z 端站点存在交集。
- 时间范围：故障发生时间相差不超过 60 天。
- “按时间”：隐藏 `in_period=false` 的历史关联记录，并按发生时间倒序。
- “按重复”：保留历史关联记录，将同一重复组放在一起，组间按最新故障时间倒序。
- 非重复故障仍可显示，但不进入重复组着色。

## 性能边界

本次明确接受明细全量返回，以恢复完整排序语义。聚合统计接口仍保持瘦身和缓存优化，不恢复在主统计响应中携带明细。只有用户进入统计页或触发下钻时，独立明细接口才返回完整结果。

## 测试

- 模板测试：四处分页 DOM 不再存在。
- 前端测试：不再存在分页状态及 `renderPagination()` 调用。
- 前端测试：物理和子公司明细缓存完整结果，并通过 `sortDetailRows()` 切换排序。
- 后端测试：两个明细接口不再切片，也不返回 `pagination`。
- 后端测试：故障明细返回 `in_period=false` 的关联历史重复故障。
- 回归测试：有效历时、业务类型映射、全部裸纤业务切换继续有效。

## 非目标

- 不改变聚合图表和 KPI 的统计口径。
- 不改变重复故障判定规则。
- 不为明细表增加虚拟滚动或懒加载。
- 不修改 NetBox 核心目录。
