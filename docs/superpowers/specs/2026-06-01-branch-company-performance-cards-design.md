# 分公司绩效考核卡片设计

## 背景

故障统计页已经有独立的“分公司” Tab，统计范围固定为六个分公司省份：浙江、山东、内蒙、陕西、四川、江西。现有页面已经提供总体情况、光缆中断情况、故障数/历时柱图、箱线图、有效平均历时、年初至今周趋势和明细下钻。

本设计目标是在现有分公司 Tab 中新增一套适合绩效考核的卡片展示。卡片参考裸纤业务故障卡片的信息层级，但不直接复制业务卡片，因为分公司考核更需要横向公平性、责任归属、扣分解释和全量影响保留。

## 目标

1. 为六个分公司各展示一张绩效考核卡片。
2. 默认展示“责任得分”，用于绩效考核。
3. 同时保留“全量得分”和全量影响指标，用于运营复盘。
4. 每张卡片必须解释扣分来源，不能只展示最终分。
5. 评分优先使用百公里归一化指标，避免线路规模差异造成不公平。
6. 保持现有分公司图表和明细下钻能力，不替换已有分析区。

## 非目标

1. 第一阶段不新增 NetBox 核心 API，也不修改 NetBox 核心目录。
2. 第一阶段不通过评论、摘要等自由文本强行识别责任归属。
3. 第一阶段不把绩效规则固化为不可配置的唯一规则。字段和权重应集中定义，便于后续调整。
4. 第一阶段不移除现有分公司图表、明细表和下钻过滤。

## 页面位置

在“分公司” Tab 顶部新增“六省绩效考核”区域，放在现有“总体情况”和“光缆中断情况”之前或两者之间。推荐顺序：

1. 六省绩效考核卡片区
2. 现有总体情况
3. 现有光缆中断情况
4. 现有图表区
5. 现有明细下钻表

这样用户进入分公司 Tab 后先看到绩效结果，再向下查看图表和明细支撑。

## 卡片布局

每张分公司卡片包含五个区域。

### 1. 标题区

标题区展示：

- 分公司名称，例如“浙江分公司”
- 责任得分，作为主视觉数字
- 等级标签
- 本期状态提示

等级建议：

| 责任得分 | 等级 | 含义 |
| --- | --- | --- |
| 90-100 | 优秀 | 本期考核表现稳定 |
| 80-89 | 良好 | 有少量扣分项 |
| 70-79 | 关注 | 需要分析原因并跟踪 |
| 0-69 | 待整改 | 需要明确整改项 |

### 2. 双口径分数

每张卡片同时展示：

- 责任得分：默认大号显示，用于绩效考核。
- 全量得分：小号辅助显示，用于运营复盘。
- 分数差异说明：当责任得分明显高于全量得分时，提示“主要影响来自非责任/降权故障”。

示例：

```text
责任 88 分 / 全量 73 分
全量影响较高，责任口径已排除或降权 5 起非责任故障
```

### 3. 核心 KPI

卡片中展示一组紧凑 KPI：

- 责任中断起数
- 责任中断总历时
- 有效平均历时
- 百公里中断起数
- 百公里中断历时
- 长时/超时/重复故障

全量 KPI 不在主区全部展开，避免卡片过载。建议以小号辅助区或 tooltip 展示：

- 全量中断起数
- 全量中断总历时
- 全量百公里起数
- 全量百公里历时

### 4. 扣分解释

卡片必须展示扣分结构：

```text
频次 -12
历时 -9
有效平均 -4
长时/超时 -6
重复 -3
趋势 -2
```

扣分项应支持点击或后续下钻。第一阶段可以先复用现有分公司明细过滤能力；例如点击“重复 -3”后筛选该分公司的重复故障。

### 5. 原因和趋势

底部展示简化分析：

- 责任口径原因 TOP3
- 全量口径原因 TOP3
- 年初至今周趋势迷你图
- 本期异常提示，例如“重复中断 2 起，较上期 +1”

趋势图不替代现有完整周趋势图，只用于卡片内快速识别。

## 评分机制

采用 100 分扣分制。默认展示责任得分，另行计算全量得分。

```text
责任得分 = 100
  - 频次扣分
  - 历时扣分
  - 有效平均历时扣分
  - 长时/超时扣分
  - 重复故障扣分
  - 趋势恶化扣分
```

全量得分使用同一公式，但输入数据使用全量故障，不应用责任排除或降权。

### 推荐权重

| 模块 | 权重 | 指标 |
| --- | ---: | --- |
| 频次 | 30 | 百公里中断起数 |
| 历时 | 25 | 百公里中断总历时 |
| 有效平均 | 15 | 过滤 0.5 小时及以下短时后的平均历时 |
| 严重度 | 15 | 长时、超时、未恢复故障 |
| 重复 | 10 | 60 天内同 A/Z 端重复中断 |
| 趋势 | 5 | 与上期或年内滚动均值相比的恶化程度 |

### 扣分建议

为避免少量异常导致分数剧烈波动，各项扣分采用上限封顶。

| 扣分项 | 建议规则 |
| --- | --- |
| 频次扣分 | 按百公里中断起数分档扣分，最高 30 分 |
| 历时扣分 | 按百公里中断总历时分档扣分，最高 25 分 |
| 有效平均扣分 | 有效平均历时超过阈值后分档扣分，最高 15 分 |
| 长时/超时扣分 | 6 小时、8 小时、12 小时、未恢复分档加重，最高 15 分 |
| 重复扣分 | 每起重复故障扣固定分，最高 10 分 |
| 趋势扣分 | 较上期或年内均值恶化时扣分，最高 5 分 |

具体阈值建议在实施前从历史数据中取分位数校准。没有稳定历史基线时，第一阶段可使用配置常量并在页面说明中标注。

## 双口径责任规则

每条故障进入以下三类之一：

| 类型 | 责任得分 | 全量得分 | 说明 |
| --- | --- | --- | --- |
| 责任计入 | 100% 计入 | 100% 计入 | 分公司维护、处置、资源质量相关 |
| 降权计入 | 例如 50% 计入 | 100% 计入 | 第三方施工、跨单位协同等可部分归责事件 |
| 仅全量计入 | 0% 计入 | 100% 计入 | 自然灾害、上级计划割接、明确非分公司责任事件 |

第一阶段可基于现有字段形成规则：

- `fault_category`
- `interruption_reason`
- `interruption_reason_detail`
- 光缆属性字段
- 是否超时
- 是否挂起
- 是否重复

如果现有字段无法稳定表达责任归属，应在数据结构中返回 `responsibility_basis` 说明“规则无法识别”，而不是从自由文本猜测。

## 后端边界

建议在 `statistics_views.py` 中拆出纯函数，便于源码级测试：

```python
def _classify_branch_fault_responsibility(fault) -> dict[str, object]:
    """Return responsibility weight, label, and basis for one branch fault."""
```

```python
def _calculate_branch_performance_score(metrics: dict[str, float]) -> dict[str, object]:
    """Return final score and deduction breakdown."""
```

```python
def _build_branch_company_performance_cards(
    branch_faults: list,
    path_lengths: dict[str, float],
    start_date,
    end_date,
    now,
) -> list[dict[str, object]]:
    """Build six branch performance cards."""
```

所有新增 Python 函数需要类型提示。第一阶段不新增数据库字段，除非后续确认现有字段无法支撑责任归属。

## 前端边界

建议新增渲染入口：

```javascript
function renderBranchCompanyPerformanceCards(cards) {
    // 渲染六省绩效考核卡片
}
```

页面 DOM 建议新增容器：

```html
<section class="statistics-branch-performance-section mb-4">
  <div id="branch-company-performance-cards" class="statistics-branch-performance-grid"></div>
</section>
```

视觉风格应延续当前统计页：

- Bootstrap 5 工具类
- 现有 `statistics-strip-card` 一类的白底、细边框、浅阴影
- 卡片圆角不超过 8px
- 指标紧凑排列
- 不使用营销式大卡片

## 交互

第一阶段交互保持轻量：

1. 点击分公司卡片标题或主分数，筛选该分公司明细。
2. 点击扣分项，叠加对应指标过滤，例如重复、超时、长时。
3. 卡片内提供责任/全量切换或小型对比，不跳转新页面。
4. 现有图表点击下钻逻辑继续可用。

## 测试计划

建议新增或扩展测试：

1. 后端源码级测试确认 `performance_cards` 返回六个分公司。
2. 后端测试确认责任得分和全量得分同时存在。
3. 后端测试确认扣分项包含频次、历时、有效平均、严重度、重复、趋势。
4. 后端测试确认百公里指标使用 `BRANCH_PROVINCE_PATH_LENGTHS`。
5. 前端源码级测试确认模板存在 `branch-company-performance-cards` 容器。
6. 前端源码级测试确认 JS 存在 `renderBranchCompanyPerformanceCards`。
7. 前端源码级测试确认分公司 Tab 加载数据后会渲染绩效卡片。
8. JS 语法检查和 `statistics_views.py` 编译检查。

## 风险与待确认

1. 责任归属字段可能不足。若现有字段无法稳定判断，应新增结构化字段后再做精确考核。
2. 阈值需要历史数据校准。直接拍固定阈值可能导致分数过松或过严。
3. 全量得分和责任得分同时出现时，需要明确默认考核口径，避免误读。
4. 降权规则要少而清晰。过多例外会削弱绩效结果的可信度。
5. 如果未来需要导出考核报表，应复用同一套后端评分函数，避免页面分数和报表分数不一致。

## 推荐实施顺序

1. 先在后端生成六省卡片数据和评分结果。
2. 用源码级测试锁定响应结构、六省完整性和扣分项。
3. 在模板加入卡片容器。
4. 在 JS 中渲染六省卡片并接入现有明细过滤。
5. 补 CSS，保持与现有统计页卡片风格一致。
6. 根据实际历史数据调整阈值配置。
