# 裸纤业务“全部业务”切换修复设计

## 背景

提交 `d093d50` 为降低故障统计接口负载，将 `ServiceStatisticsDataAPI` 改为默认只返回当前统计周期内发生故障的业务。前端仍在已有响应数据中切换“发生故障业务”和“全部业务”，因此两种状态实际展示相同。

## 目标

- 默认进入裸纤业务页时仍只加载当前周期发生故障的业务，保留现有性能优化。
- 首次切换到“全部业务”时按需加载全部裸纤业务。
- 同一统计条件下重复切换不产生重复请求。
- 更改统计周期后重新按当前切换状态加载正确数据。
- 电路业务页和业务故障明细分页行为保持不变。

## 接口设计

复用现有 `statistics/service-data/` 接口，增加可选查询参数：

```text
include_all_bare_fiber=1
```

未提供该参数时，接口保持当前行为，只返回本期有故障影响的业务。

提供该参数时：

1. 预先查询全部 `BareFiberService`，以零故障统计结构初始化裸纤业务卡片。
2. 当前周期内有影响的裸纤业务继续叠加真实统计数据。
3. 电路业务仍只返回当前周期有故障影响的业务。
4. 无故障裸纤业务的当前周期统计为零，SLA 为 100%，年度和日历结构保持前端所需格式。

## 前端数据流

前端维护当前裸纤业务响应是否包含全量业务的状态。

- 初始进入或选择“发生故障业务”：请求默认接口。
- 首次选择“全部业务”：使用 `include_all_bare_fiber=1` 重新请求聚合接口。
- 全量请求完成后缓存该统计条件下的服务数组，后续在两个选项间只执行本地过滤和重绘。
- 日期或统计周期变化时清除缓存；如果当前选择为“全部业务”，新请求直接携带全量参数。
- 请求失败时保留当前卡片并展示现有加载失败状态，不将缓存标记为已完成。

为避免快速切换或周期变更导致旧请求覆盖新数据，请求结果只在其统计条件仍与当前页面一致时应用。

## 测试

后端回归测试锁定：

- 默认请求不初始化全部裸纤业务。
- `include_all_bare_fiber=1` 时查询并初始化全部裸纤业务。
- 电路业务不会因该参数扩展为全量。

前端源码回归测试锁定：

- 全量切换会触发带参数的请求。
- 已加载全量数据后切换仅本地重绘。
- 统计条件变化会使全量缓存失效。
- 默认“发生故障业务”行为保持不变。

验证范围包括定向统计测试、JavaScript 语法检查和 Python 编译检查。

## 非目标

- 本次不新增独立的全部业务分页接口。
- 本次不调整业务故障明细分页接口。
- 本次不改变裸纤业务和电路业务的统计口径。
