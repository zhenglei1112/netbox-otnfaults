# 分公司绩效多维度指标与业务加权考核方案设计 PRD

## 1. 业务背景与痛点分析

在现有的 OTN 故障统计及绩效评估体系中，分公司的核心绩效考评基于“物理资产规模”进行归一化，即以**百公里光缆故障频次**（`count_per_100km`）和**百公里故障历时**（`duration_per_100km`）作为核心责任扣分项。然而，在实际运行和精细化运维管理中，这种纯物理维度的考核方式暴露出了严重的“逆向淘汰”和“公平性缺失”痛点：

1. **业务量差异巨大，导致“干得多错得多，不干活躺赢”**：
   * **骨干/大省份**（如浙江、江西）：承担着极其繁重的网络枢纽和海量跨省业务（承载业务量可达数千甚至上万条）。即使运维水平极高，由于物理路由复杂、外部建设施工频繁，故障的绝对数量客观上仍然偏高。一旦发生物理阻断，影响的业务数量巨大，在绩效上很容易被判定为“待整改”或“关注”。
   - **偏远/新起步省份**（如内蒙、四川）：承载的业务量极少（甚至为 0 业务）。网络结构极度简单，几乎不会发生故障。在当前考核体系下，这些省份长年获得 **100 分满分（优秀）**。
   * **矛盾冲突**：这导致实际承担核心产值、面临极高运维风险的大省在绩效上反而落后于没有业务负担的边缘小省，打击了核心运维团队的积极性。

2. **容灾抗灾成效被物理指标掩盖**：
   * 某些分公司投入巨大精力和资金进行了 **1+1 双路由保护或 ASON 智能自愈网络建设**。即使某段光缆发生 5 次物理阻断，业务依然能通过备用路由瞬间倒换，**对最终客户造成的实际业务受阻为 0**。
   * 在纯物理指标考核中，这类具有优秀容灾水平的分公司由于“物理发生次数多”依然面临高额扣分；而没有做任何容灾保障、仅发生 1 次阻断但导致核心业务中断 20 小时的省份，反而因为物理次数少而绩效得分更高。这产生了极其错误的管理导向。

3. **SLA 业务价值未作区分**：
   * 现有的故障扣分对“特级/一级（如重要政企、生命通道业务）”与“三级（普通宽带、内部测试）”业务的中断未作区分。无法在绩效上引导运维资源向高价值业务倾斜。

---

## 2. 核心设计原则

为了解决上述痛点，本方案遵循以下四大原则：
* **业务导向（Service-Centric）**：把“客户感知与业务连续性（SLA）”作为衡量分公司绩效的最核心考量，而非仅仅看冰冷的光缆断裂次数。
* **业务量稀释效应（Volume Dilution）**：引入“业务保障率”等以业务量为分母的指标，使海量业务的保障功劳能够客观稀释小额不可抗力故障的扣分。
* **物理与业务双轨制（Dual-Track Balance）**：既要考核“物理线路维护质量（百公里考核）”，也要考核“承载业务连续性保障（业务量加权考核）”，两手都要硬。
* **小省负荷下限兜底（Minimum Load Floor）**：防止 0 业务或极低业务省份通过“零故障躺赢”占据优秀等级，对其设置考核系数修正或参与能级分级。

## 3. 建议引入的横向对比新指标

除了现有的物理指标，我们建议在分公司统计页横向对比中，增加以下核心指标进行横向雷达图或卡片对比：

```mermaid
graph TD
    A[分公司绩效多维度考核体系] --> B[物理维护维度 (消减规模劣势)]
    A --> C[业务保障维度 (引入业务量加权)]
    A --> D[运维响应维度 (衡量管理效能)]

    B --> B1[百公里故障频次]
    B --> B2[百公里故障历时]
    B --> B3[重复发生率 (60天相同站点)]

    C --> C1[SLA 业务可用率]
    C --> C2[百条业务中断频次]
    C --> C3[单次故障平均影响业务数]
    C --> C4[加权中断当量]

    D --> D1[平均抢修历时 (MTTR)]
    D --> D2[故障主动发现率]
    D --> D3[光缆隐患整治率]
```

### 3.1 业务保障指标（SLA维）

#### 1. SLA 业务可用率 (Service Availability / SLA Rate)
* **数学定义**：
  $$A = 1 - \frac{\sum_{i=1}^{N} (S_i \times T_i)}{N \times H}$$
  其中：
  * $N$：分公司承载的总业务数（如 `BareFiberService` 数量）。
  * $S_i$：第 $i$ 条业务在统计周期内的受阻状态（1为受阻，0为正常）。
  * $T_i$：第 $i$ 条业务的受阻时长（单位：小时）。
  * $H$：统计周期总小时数（如单月为 720 小时）。
* **应用意义**：衡量分公司所有承载业务的整体稳定度。由于大省的 $N$ 极大，其单个业务的故障会被庞大的总业务基数稀释，体现了“大体量、高容错”的客观规律，非常公平。

#### 2. 百条业务中断频次 (Outages per 100 Services)
* **数学定义**：
  $$O_{100} = \frac{\text{周期内发生业务中断的故障次数}}{\text{分公司总承载业务数}} \times 100$$
* **应用意义**：相比于直接统计故障次数，百业务中断频次使小省的故障扣分权重极大提升，而大省因为分母大获得了合理的免责缓冲。

#### 3. 单次故障平均影响业务数 (Average Services Impacted per Fault)
* **数学定义**：
  $$\overline{I} = \frac{\text{受阻业务总数}}{\text{物理故障发生次数}}$$
* **应用意义**：**直接衡量网络的结构健壮性（如保护路由和 ASON 的生效情况）**。若该指标接近于 0，即使物理故障多，也说明分公司容灾方案做得极其完美，绩效上应给予**免扣分甚至额外加分**。

---

### 3.2 运维效能指标（管理维）

#### 4. 平均抢修历时 (MTTR - Mean Time to Repair)
* **数学定义**：
  $$\text{MTTR} = \frac{\text{光缆故障总阻断历时}}{\text{物理故障发生次数}}$$
* **应用意义**：排除业务多寡、光缆长短的干扰，专门考核分公司应急抢修队在发生物理阻断时的**纯响应速度和熔接效率**。无论大省小省，拼的就是抢修硬功夫。

#### 5. 故障主动发现率 (Proactive Detection Rate)
* **数学定义**：
  $$P_{proactive} = \frac{\text{通过网管告警/光路监测主动派单的故障数}}{\text{总故障数}} \times 100\%$$
* **应用意义**：评估运维的主动防御能力。凡是主动发现并处理的故障，由于避免了客户先投诉，应在扣分时享有 50% 的豁免。

## 4. 业务量差异对绩效考核的量化模型设计

为彻底消除“0业务躺赢，业务多躺枪”的现象，设计以下三种绩效量化模型供考核委员会选择或结合使用：

### 4.1 方案 A：SLA 负荷豁免额度制 (SLA Allowance Model) —— 【推荐首选】

> [!TIP]
> **设计思想**：参照云计算领域的保障机制，根据分公司承担的“业务总负荷”（业务条数 × 统计周期总时间），按万分之一（或其他保障水平）赋予分公司一个**“免扣分受阻时长额度（业务·小时）”**。在该额度内的故障受阻，只记录不扣分；超出额度后才启动绩效扣分。

#### 1. 额度计算公式：
$$\text{免费豁免额度 } E_{free} = N_{service} \times H_{period} \times \alpha$$
其中 $\alpha$ 为行业故障容忍系数（例如，要求业务可用率在 99.98% 以上，则可容忍系数 $\alpha = 0.02\%$）。

*   **实例对比 (单月 720h, $\alpha = 0.02\%$)**：
    *   **大省份（江西：3500 条业务）**：
        $$E_{free} = 3500 \times 720 \times 0.0002 = 504 \text{ 业务·小时}$$
        *   江西单月若发生 3 次光缆中断，每次波及 20 条业务，平均抢修 4 小时，总受阻量为 $3 \times 20 \times 4 = 240$ 业务·小时。
        *   由于 $240 < 504$，**江西在业务保障维度获得 0 扣分（满分）**！完美保护了保障巨量业务的功劳。
    *   **小省份（内蒙：10 条业务）**：
        $$E_{free} = 10 \times 720 \times 0.0002 = 1.44 \text{ 业务·小时}$$
        *   内蒙由于业务极少，一旦发生 1 次故障波及 1 条业务，只要中断超过 1.5 小时，其受阻量即达到 2.0 业务·小时，超出额度，**触发扣分**。
*   **管理效果**：极其科学。大省的小规模随机物理故障被负荷豁免抵消；小省因为业务极少，容错率变得极低，逼迫其对极少的核心业务投入 100% 的精力，告别“因为业务少而粗心大意”的情况。

---

### 4.2 方案 B：双轨制权重划分与 0 业务惩罚 (Dual-Track weighting with Zero-load Penalty)

> [!IMPORTANT]
> **设计思想**：将分公司绩效分为“物理维护”与“业务保障”两个平行轨道，分别打分后加权。同时，对 0 业务或业务量低于门槛值的省份，降低其绩效总分上限或设置加权惩罚，使其无法通过“躺平”获得优秀。

#### 1. 权重模型：
$$\text{绩效总得分 } S_{total} = (S_{physical} \times W_{phy}) + (S_{service} \times W_{srv})$$

*   对于**正常业务省份**：$W_{phy} = 40\%$，$W_{srv} = 60\%$（体现业务导向）。
*   对于**0 业务或低于门槛（如少于 20 条业务）的省份**：
    *   **惩罚机制**：其业务保障分 $S_{service}$ 无法被计为 100 分。该省份的业务保障分强制按“全网平均业务保障分”或“固定 70 分（合格线）”计入，或者直接将 $W_{srv}$ 减半，并将总分上限封顶在 85 分（即最高只能评为“良好”，彻底剥夺“优秀”资格）。
    *   只有当分公司积极拓展业务、承载业务量突破门槛后，才能激活高绩效通道。
*   **管理效果**：直接杜绝 0 业务省份躺赢。不承担集团核心业务的分公司，即使物理指标再好，也只能拿个及格或良，逼迫各分公司主动向集团申请承载更多骨干路由和业务分流，推动分公司在业务发展上的积极性。

---

### 4.3 方案 C：业务重要度级加权 (SLA Grade Matrix Weighting)

> [!CAUTION]
> **设计思想**：并非所有业务价值均等。大省承担的大多是核心骨干、特级/一级的高产值业务，而小省可能只是边缘的测试链路。必须在绩效考核中体现“业务重要度加权中断当量”。

#### 1. 权重定义：
*   **特级业务 (Special Grade)**：权重 $w = 10.0$
*   **一级业务 (Grade 1)**：权重 $w = 5.0$
*   **二级业务 (Grade 2)**：权重 $w = 2.0$
*   **三级业务 (Grade 3)**：权重 $w = 1.0$

#### 2. 加权受阻当量计算：
$$\text{加权受阻当量 } WO = \sum_{j} (w_j \times T_j)$$
（$w_j$ 为第 $j$ 个中断业务的等级权重，$T_j$ 为该业务中断的小时数）。

#### 3. 业务绩效得分公式：
$$S_{service} = \max\left(0, 100 - \frac{WO}{\text{本省业务总加权基数}} \times 扣分系数\right)$$
其中 $\text{本省业务总加权基数} = \sum_{k=1}^{N} w_k$。
*   **管理效果**：由于大省的业务结构中高等级业务占比大、基数极其雄厚，如果分公司将 high 等级业务保护得极好，其“业务总加权基数”分母会非常巨大，从而使得个别低等级业务的偶然中断扣分微乎其微。这正向引导分公司**“优先抢修保障核心业务，对不同客户实行阶梯化运维”**。

## 5. 分公司统计页前端 UI 落地构想

为了在分公司统计大屏/页面上直观、震撼地呈现这些新指标，我们建议进行以下 UI 升级：

1. **“业务量-故障率”双轴散点气泡图（Matrix Bubble Chart）**：
   * **横轴 (X)**：承载业务量（条）。
   * **纵轴 (Y)**：物理故障频次（次/百公里）。
   * **气泡大小**：代表该分公司的总路程公里数（`path_length`）。
   * **气泡颜色**：代表当前的绩效等级（优秀为翠绿，良好为海蓝，关注为琥珀，待整改为玫瑰红）。
   * **视觉体验**：用户可以一眼看出：
     * 处于**右下角**（业务量大、故障率低）的省份是“明星保障分公司”（如江西）。
     * 处于**左下角**（业务量极少、故障率低）的省份是“低负荷稳健分公司”（如内蒙），气泡很小。
     * 处于**左上角**（业务少、故障多）的省份是“急需整改分公司”（如四川）。
     * 这种对比极大提升了高体量分公司的荣誉感，并且让决策者对各省的真实负荷与运维质量一目了然。

2. **分公司绩效卡片中的“业务负荷保障”专属区域**：
   * 在绩效卡片中，除原有的“责任扣分”物理指标外，新增 **“业务连续性表现 (SLA & Load)”** 子面板：
     * **SLA 业务可用率**：显示如 `99.995%`，并带有圆环进度条（达成率）。
     * **SLA 豁免额度剩余**：进度条展示 `剩余 264h / 共 504h`。
     * **网络结构自愈率**：展示比如 `85.4%`（表示 85.4% 的物理故障被路由保护完全自愈，未造成客户中断），作为分公司基建政绩的重要支撑。
