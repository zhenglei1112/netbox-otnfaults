# 【PRD】OTN运行态势大屏 - 自适应弹性布局设计规范

## 1. 设计背景与核心原则

### 1.1 问题陈述

当前大屏（`dashboard.html`）布局为静态结构，割接和重保模块始终占据固定空间。当割接或重保无数据时，页面出现大块空白，显得系统加载异常；当数据过多时，列表无限堆叠，变成报表页。

### 1.2 核心设计原则

> **当前故障始终占主位**。大屏第一优先级永远是：当前是否有故障、故障在哪、影响什么业务。

- **有数据时**：突出风险和计划
- **无数据时**：不空、不塌、不显得系统异常
- 割接和重保作为 **可伸缩辅助信息层**，不抢占主信息层空间
- 地图和当前故障列表始终保持稳定存在

### 1.3 信息优先级层次

| 优先级 | 信息层     | 说明                     |
| ------ | ---------- | ------------------------ |
| P0     | 当前故障   | 始终显示，最高优先级     |
| P1     | 网络地图   | 始终显示，可自适应放大/缩小 |
| P2     | 割接计划   | 可伸缩，有数据显示列表，无数据显示状态卡 |
| P3     | 重保任务   | 可伸缩，有数据显示卡片，无数据显示一行状态 |
| P4     | 辅助指标   | 网络健康度、告警趋势、链路状态等补充信息 |

## 2. 割接信息模块状态设计

割接信息共分为 4 种显示状态，根据数据量自动切换。

### 2.1 状态矩阵

| 数据量   | 显示方式    | 设计重点                               |
| -------- | ----------- | -------------------------------------- |
| 无割接   | 小型空状态卡片 | 表达"近期暂无割接计划"，不能显得系统没加载出来 |
| 1-3条    | 卡片式展示  | 每条割接显示时间、地点、状态、影响范围   |
| 4-8条    | 紧凑列表    | 优先显示最近7天/48小时内割接             |
| 8条以上  | 分组 + 滚动 | 按时间窗口、区域、状态聚合展示           |

### 2.2 无割接状态

**视觉处理**：
- 保留一个小卡片，占原模块 **30%-40% 高度**
- 同时将地图或故障区域适当放大

**文案规范**（不要只写"暂无数据"）：
```
推荐文案 A：未来7天暂无割接计划
推荐文案 B：网络变更窗口稳定，未检测到待执行割接任务
```

### 2.3 数据过多状态（>8条）

右侧显示聚合摘要 + 滚动明细：
```
未来24小时：3条
未来7天：12条
待审批：2条
执行中：1条

[下方滚动最近/最重要的5条]
```

**地图聚合规则**：
- 割接点很多时使用城市级聚合：北京 3、广州 5、成都 2
- 不要所有点都展开标签
- 只展开最近24小时或高影响割接的标签

## 3. 重保信息模块状态设计

重保信息一般是文字类，不需要地图，适合做成 **底部横向信息带** 或 **右下角信息卡片**。

### 3.1 状态矩阵

| 数据量   | 显示方式    | 设计重点                       |
| -------- | ----------- | ------------------------------ |
| 无重保   | 安静型提示  | "近期暂无重保任务"，不要抢视觉  |
| 1-2条    | 横向卡片    | 显示名称、时间、保障对象        |
| 3-5条    | 底部列表    | 每条一行，突出时间范围          |
| 5条以上  | 滚动播报/分组 | 按"进行中/即将开始/已结束"分类  |

### 3.2 无重保状态

**文案规范**：
```
推荐文案 A：近期暂无重保任务
推荐文案 B：当前网络按常规运行策略保障
大屏风格 A：重保任务：0 — 当前无专项保障任务，网络运行保持常规监控
```

> ⚠️ 不要把重保模块完全隐藏（`display: none`），否则页面结构跳变，看起来像模块加载失败。

### 3.3 数据过多状态（>5条）

底部显示聚合 + 轮播：
```
进行中 2 | 即将开始 4 | 已完成 6
[下方自动轮播重点重保任务]
```

## 4. 五种布局状态设计

### 4.1 状态一：割接、重保都有数据（标准满屏）

```
┌──────────────────────────────────────────────┐
│               顶部：核心指标                   │
├──────────┬────────────────────┬───────────────┤
│ 左侧：   │  中间：地图         │ 右侧：       │
│ 当前故障  │  + 故障点           │ 未来割接计划  │
│ 列表     │  + 割接点           │              │
├──────────┴────────────────────┴───────────────┤
│               底部：近期重保信息               │
└──────────────────────────────────────────────┘
```

### 4.2 状态二：无割接，有重保

割接模块收缩，地图放大，右侧补充辅助信息。

```
┌──────────────────────────────────────────────┐
│               顶部：核心指标                   │
├──────────┬────────────────────┬───────────────┤
│ 左侧：   │  中间：地图放大     │ 右侧上：     │
│ 当前故障  │  (无橙色割接点,    │ 暂无割接提示  │
│ 列表     │   只保留红色故障点) │ 右侧下：     │
│          │                    │ 网络健康度/   │
│          │                    │ 重点链路状态  │
├──────────┴────────────────────┴───────────────┤
│               底部：近期重保信息               │
└──────────────────────────────────────────────┘
```

### 4.3 状态三：有割接，无重保

重保区域收缩为底部小提示，底部可补充辅助信息。

```
┌──────────────────────────────────────────────┐
│               顶部：核心指标                   │
├──────────┬────────────────────┬───────────────┤
│ 左侧：   │  中间：地图         │ 右侧：       │
│ 当前故障  │                    │ 未来割接计划  │
│ 列表     │                    │              │
├──────────┴────────────────────┴───────────────┤
│ 暂无重保任务 + 告警趋势/运行指标/链路利用率     │
└──────────────────────────────────────────────┘
```

底部补充信息选项：
- 近24小时告警趋势
- 网络健康度
- 链路利用率
- 重点业务在线率

### 4.4 状态四：割接、重保都没有（运行总览模式）

> 这是最容易设计失败的状态。需变成"运行总览模式"。

```
┌──────────────────────────────────────────────┐
│             网络健康状态                       │
├──────────┬────────────────────┬───────────────┤
│ 左侧：   │  中间：地图最大化   │ 右侧：       │
│ 当前故障  │                    │ 重点节点状态  │
│ 如无故障  │                    │ / 告警趋势   │
│ 则显示   │                    │              │
│ 运行概况  │                    │              │
├──────────┴────────────────────┴───────────────┤
│         运行指标 + 最近巡检信息                 │
└──────────────────────────────────────────────┘
```

**设计要点**：
- 页面重点从 **"事件展示"** 切换为 **"健康态势展示"**
- "无数据"变成一种正向状态，而不是空白状态

**推荐文案组合**：
```
未来7天暂无割接计划
近期暂无专项重保任务
当前网络处于常规运行监控状态
```

### 4.5 状态五：割接或重保数据很多

大屏不能无限堆列表，否则会变成报表页。采用 **"摘要 + 滚动明细"** 策略。

详见 §2.3 和 §3.3 的聚合显示规则。

## 5. 弹性模块规则总结

### 5.1 统一自适应规则

| 模块     | 规则                                                       |
| -------- | ---------------------------------------------------------- |
| 当前故障 | 始终显示，最高优先级                                         |
| 地图     | 始终显示，可根据其他模块数据多少自动放大/缩小                  |
| 割接     | 有数据显示列表，无数据显示小型状态卡，数据多则滚动              |
| 重保     | 有数据显示底部卡片，无数据显示一行状态，数据多则轮播            |

### 5.2 设计目标

同时兼顾三种情况：
1. **没数据时不空、不尴尬**
2. **少量数据时清晰、美观**
3. **大量数据时不拥挤、仍然可读**

---

## 6. 推荐设计风格

### 6.1 最终推荐

采用 **"方案4 GIS地图中心风格 + 方案1 深蓝指挥中心质感"** 组合。

**理由**：
- GIS地图中心风格最适合 OTN 业务场景，割接少或无割接时地图天然可以放大
- 深蓝指挥中心质感正式专业，适合运维场景
- 重保作为底部文本带，不影响主视觉
- 既适合运维场景，又不会显得过度炫技

### 6.2 五种风格的弹性适配要点

| 风格         | 无数据时的处理                                                     |
| ------------ | ----------------------------------------------------------------- |
| 深蓝指挥中心 | 割接和重保模块收缩成状态卡，地图和当前故障区域扩大                     |
| 浅色商务     | 轻量空状态卡"暂无割接计划"，视觉干净不压迫                           |
| 赛博霓虹     | 发光状态条"NO CUTOVER PLAN / 常规运行中"，但不要太夸张                |
| GIS地图中心  | 地图天然放大，重保作为底部文本带                                      |
| 工业告警驾驶舱 | 补充运行指标、告警趋势、链路利用率，保持"监控感"                     |

## 7. 技术实现要点

### 7.1 布局结构（推荐最终方案）

```
主地图 + 可伸缩右侧计划栏 + 可伸缩底部重保栏
```

### 7.2 前端自适应逻辑

需要在 `dashboard_app.js` 的数据刷新回调中，根据数据量动态调整布局：

```javascript
// 伪代码：根据数据状态决定布局模式
function determineLayoutMode(data) {
    const hasFaults = (data.active_faults || []).length > 0;
    const hasCutovers = (data.cutovers || []).length > 0;
    const hasHeavyDuties = (data.heavy_duties || []).length > 0;
    const cutoverCount = (data.cutovers || []).length;
    const heavyDutyCount = (data.heavy_duties || []).length;

    if (hasCutovers && hasHeavyDuties) return 'STATE_FULL';        // 状态一
    if (!hasCutovers && hasHeavyDuties) return 'STATE_NO_CUTOVER';  // 状态二
    if (hasCutovers && !hasHeavyDuties) return 'STATE_NO_HEAVY';    // 状态三
    if (!hasCutovers && !hasHeavyDuties) return 'STATE_OVERVIEW';   // 状态四
}

// 割接模块显示模式
function getCutoverDisplayMode(count) {
    if (count === 0) return 'empty-card';       // 小型空状态卡
    if (count <= 3) return 'card-list';         // 卡片式展示
    if (count <= 8) return 'compact-list';      // 紧凑列表
    return 'grouped-scroll';                    // 分组+滚动
}

// 重保模块显示模式
function getHeavyDutyDisplayMode(count) {
    if (count === 0) return 'quiet-hint';       // 安静型提示
    if (count <= 2) return 'horizontal-cards';  // 横向卡片
    if (count <= 5) return 'bottom-list';       // 底部列表
    return 'carousel-grouped';                  // 滚动播报/分组
}
```

### 7.3 涉及修改文件清单

| 文件                         | 修改内容                         |
| ---------------------------- | -------------------------------- |
| `dashboard.html`             | 调整HTML结构支持弹性布局          |

## 8. 与现有代码的关系

- 当前 `dashboard.html` 布局：左翼面板 → 地图 → 右翼面板 → 底部Ticker
- 当前右翼面板包含：综合事件队列 + 焦点详情
- 当前底部：运行事件 Ticker 滚动条
- 当前重保：地图顶部跑马灯横幅（`heavy-duty-banner`）

**改造方向**：
- 右翼面板需支持"割接计划列表"与"空状态卡"的切换
- 底部 Ticker 需支持在无重保时变为辅助信息展示
- 地图容器需支持根据右侧/底部模块伸缩而动态调整大小
- 重保横幅需从纯跑马灯升级为支持多种显示模式
