# 故障图标动态外圈渲染方案设计

为故障分布图增加故障点（特别是进行中故障）的“呼吸/波纹”动态外圈效果，考虑现有的 `MapLibre GL` 与 `Deck.gl` 以及 Canvas 图标渲染逻辑，设计了以下四种方案：

## 背景分析
目前系统在 `fault_mode.js` 中使用 `Canvas` 将 `SVG` 预渲染为静态 `ImageData`(64x64)，再通过 MapLibre 的 `map.addImage` 注入内存。地图使用 `symbol` 图层进行渲染，并利用了内置的插值实现随缩放自动适配尺寸。

---

## 方案对比

### 方案一：Canvas 动画帧结合 `map.updateImage` (✅ 最推荐)

**实现思路**：
利用原有的 `Canvas` 渲染机制，将对应关键状态（如 `processing` 处理中状态）的图标独立出一个实时动画循环（`requestAnimationFrame`）。每帧在这个 64x64(或更大，如 96x96，以便画出外圈) 的 Canvas 会布尔计算并重绘中心静态图标，外加一个不断扩大的透明度渐变同心圆。画完后执行 `map.updateImage(iconName, imageData)` 更新渲染内容。

**优势**：
- **契合技术栈**：完全复用地图原有的 `symbol` 类型和 `icon-size` 自动缩放策略。不需要增加额外图层。
- **性能优越**：因为同一种状态的图标共用同一份图片材质，不管地图上有多少个点，CPU 一帧只需要重画一张图片片段，MapLibre 的 WebGL 会自动应用到所有属于该图标的点，渲染流畅度极高。
- **维护成本低**：只需修改 `fault_mode.js` 里的 `createColoredIcon`，为特定状态名注册一个动画帧回调即可。

**劣势**：
- 动态图像必须留出边缘透明空白用于画波纹（比如要生成大一点的图像边界防止截断），会略微改变基准 Icon 的布局尺寸设计。

### 方案二：新增 MapLibre 原生 `Circle` 图层进行属性渐变动画

**实现思路**：
新添加一个 MapLibre GL 的图层（`type: 'circle'`），图层数据源同故障点数据。我们在 JS 代码里启一个定时循环，不断计算并修改图层的半径和透明度，并通过内置的 `map.setPaintProperty` 方法实时下发新的配置。

**优势**：
- **逻辑清晰**：独立了波纹与点的逻辑图层，不会侵入现在的符号图标 Canvas 处理逻辑。

**劣势**：
- **掉帧隐患**：不断调用 `setPaintProperty` 触发引擎重算样式数据包，开销大于单次更新单个材质图片缓存，如果数据多容易卡顿掉帧。
- **缩放协调度**：需要自行手动模拟随 Zoom 缩放的插值计算去对应原图标的大小，极其繁琐。

### 方案三：使用 Deck.gl `ScatterplotLayer` 动画或自定义 Shader

**实现思路**：
利用项目现有的 `Deck.gl` 能力（已存在 `DeckGLFlowAnimator`），使用 DeckGL 加一层透明的 `ScatterplotLayer`。配合内部提供给着色器的 `time` 作为 uniform ，直接在 GLSL 着色器级别实现一个动态波纹。

**优势**：
- **极致性能**：动画与透明度计算完全在显卡中完成，适合万级以上元素海量呈现，无需主线程 JS 参与计算。

**劣势**：
- 技术要求极高（需要深入 WebGL/GLSL 以及 DeckGL 自定义图层系统）。
- 与 MapLibre 的原图层有时因为同步差异在急速平移时呈现微弱剥离感。如果节点总量在几百以内，属于性能“杀鸡用牛刀”，增加了复杂的心智负担。

### 方案四：传统的 HTML/CSS DOM Marker (`maplibregl.Marker`)

**实现思路**：
避开图层渲染，将动态要素改为以原生 DOM 元素插入地图，并使用纯 CSS 的 `@keyframes pulse { 0% {box-shadow...}}` 进行扩散渲染。

**优势**：
- 前端实现最容易，各种光波特效写 CSS 即可完成。

**劣势**：
- **性能最差**：如果地图当前视野内元素超过上百个，会导致 DOM 重绘卡顿，脱离 WebGL 渲染流程。
- **不支持图层特性**：无法做到原生 Label、点位间的自动避障隐藏。

---

## 综合结论与反馈

**强烈建议选择【方案一：Canvas 帧结合 `map.updateImage`】**。
核心原因：利用现有框架特性发挥四两拨千斤的作用。故障数量即使上千，在地图上也不过是复用同一个缓存的 `<canvas>` 图片。修改起来十分优雅且性能非常强大。
